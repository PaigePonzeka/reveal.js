Sass is a CSS preprocessor

Sass is a CSS preprocessor. It's a scripting language that extends CSS. I like to think that it makes writing CSS not suck. Sass Gives CSS the power of nesting, variables, functions, mixins, inline partials. And those features let you write some pretty cool shit.

Nesting

Nesting in Sass makes your CSS easier to read and maintain. It gives you a visual hierarchy. So you can go from something like this table css. To this Table.scss.

The Inception Rule

Don't go more than four levels deep

Be careful to limit your nests. Generally speaking you should nest over 4 times otherwise that will result in overqualified CSS that will be hard to override and maintain.

& - Ampersand

Ampersand is used when nesting. It tells Sass to pull down any parent selector into the child selector. Using inline imports we can organize our SCSS files into partial files just as we do with our html templates. Partials are generally, denoted with an underscore. You'd have to manually comb through hundreds and maybe thousands of lines of code.

This actually happened

So this actuallly happened before. We were using Sass at the time but we didn't assign any of our colors to varialbles so I had to do just that and Comb through thousands of lines of CSS, find every instance of a color and change it. I spent a pretty long time changing one shade of gray to a different shade of gray. It was wonderful. But I made sure when I did it I added variables to our color assignments. You can see the colors in the _color.scss files. Pictured here.

Naming Conventions

Also Note that I named the variables to include their colors names. That wouldn't be useful if we needed to change the colors later, since it would be confusing if $brand-blue was actually gray later on. Instead I made them colors that were obvious to a user what they could be. So for usability to other developers initial variables are names after their colors. But then when they are used in CSS for example $color-location-cms is to indicate the color is it on the navigation bar and even without being familar with storm I can know its some shade of blue.

And if we change again?

If Howard ever feels that itch of inspiration again. We can change all of the colors of storm but just changing a few lines in the _color.scss partial and deploying are changes. As long as everyone else working in the SASS had been using the color variables.

Sass Operations

Sass has some built in helper math operations that can be very useful. You can find the documention at sass-lang.com. It can do basic mathematical operations such as addition, subtraction, division. There's also some number functions, String functions and color functions that can be helpful say if you wanted to lighten or darken a color for an accent. You can just pass in a color an a percentage and it will darken it for you. I actually use those one's a lot.

@extend

Let's take a break from refactoring to take a look at another powerful Sass feature. @extend is another nifty directive in Sass. It allows you to share rules between selectors and it groups extend selectors together.

We use @extend heavily in the Storm buttons

You can see it in action here in the _buttons partial. You might ask yourself why do you have btn-orange and btn-blue if all of the storm buttons are gray. Well remember that 50 Shades of Grey scenario i mentioned earlier? An old branding of Storm had different colored buttons for different actions. When we rebranded we got rid of those buttons and shifted to all gray buttons (minus the warning button). To keep compatibility with old styles and to potentinally keep from breaking any javascript that might happen to use those styling classes for their code I kept the classes around and just inherited style from other buttons. (Also for the record, never use a styling class as a JQuery selector, whether or not I want a dark gray or light gray button shouldn't also affect whether the buttons runs an ajax call or not)

What can we do about this?

There's one fix it to just turn the shared styling into a mixin and include it on both classes. That will duplicate the CSS and put it in bother places. There's another way we can do it with the @extend directive.And luckily for us. The creators of Sass introduced a new directive to address this very problem. Meet Placeholders.

%Placeholders

Instead of using a period infront of the extended class, you can use the percent sign and that prevents the selector from being inserted into the CSS.
